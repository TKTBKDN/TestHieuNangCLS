📋 KỊCH BẢN KIỂM THỬ HIỆU NĂNG HỆ THỐNG LMS

Dự án: Kiểm thử hiệu năng hệ thống đào tạo trực tuyến (LMS)
Công cụ: Grafana k6
Mục tiêu: Load Test (1.000 / 2.000 / 7.000 CCU) & Stress Test
Ngày cập nhật: 22/05/2026

1. MỤC TIÊU KIỂM THỬ (Theo yêu cầu khách hàng)

  1. Load Test (Kiểm thử tải):
    • Đánh giá khả năng đáp ứng và đo lường tài nguyên tiêu hao (CPU/RAM/Network) của hệ thống ở các mốc: 1.000 CCU, 2.000 CCU và 7.000 CCU.
  2. Stress Test (Kiểm thử giới hạn):
    • Tăng tải vượt ngưỡng 7.000 CCU để tìm điểm gãy (Breakdown Point) — thời điểm hệ thống bắt đầu quá tải, phản hồi cực chậm hoặc sập hoàn toàn (trả lỗi 502/504).

📐 QUY ĐỔI VU ↔ CCU

VU (Virtual User) trong k6 là luồng ảo chạy liên tục không ngừng, khác với CCU (Concurrent User) là người dùng thật có thời gian đọc, suy nghĩ (~15 giây) giữa các thao tác.

VU = CCU ÷ (Think-time thực tế ÷ Sleep trong k6)
Với Think-time ≈ 15s, Sleep k6 = 1s → 1 VU ≈ 15 CCU
CCU mục tiêu VU cấu hình k6 (sleep=1s) Giải thích
1.00067 VU67 × 15 ≈ 1.000 CCU
2.000134 VU134 × 15 ≈ 2.000 CCU
7.000467 VU467 × 15 ≈ 7.000 CCU

Kịch bản tập trung kiểm thử 2 API cốt lõi của quy trình thi trắc nghiệm:

STT Nghiệp vụ API Endpoint Phương thức Mô tả tải
1 Bắt đầu thi /api/examtest/start-exam PATCH Khởi tạo bài thi trắc nghiệm — API chịu tải cao nhất
2 Nộp bài thi /api/examtest/set-answer-question POST Nộp đáp án từng câu hỏi — ghi dữ liệu vào Database

3. CẤU HÌNH CÁC KỊCH BẢN CHẠY TEST

Để đạt được mục tiêu tính toán tài nguyên và tìm điểm gãy, quá trình test được chia làm các bước độc lập sau:

📊 Kịch bản 3.1: Load Test 1.000 CCU (Tính toán tài nguyên mốc 1)
Số lượng VU:
67 VU (67 × 15 ≈ 1.000 CCU)
Iterations:
1.000 lượt (mỗi user thực hiện 1 lượt đầy đủ)
Sleep:
1 giây (mô phỏng think-time người dùng thật)
API:
Bắt đầu thi + Nộp bài thi
Mục tiêu: Đo CPU/RAM/Băng thông trên máy chủ Web và Database khi phục vụ ổn định 1.000 người cùng lúc.
📊 Kịch bản 3.2: Load Test 2.000 CCU (Tính toán tài nguyên mốc 2)

Iterations:
2.000 lượt
Sleep:

Mục tiêu: Đánh giá hiệu năng hệ thống khi tải tăng gấp đôi so với mốc 1. Ghi nhận mức tăng CPU/RAM tương ứng.
📊 Kịch bản 3.3: Load Test 7.000 CCU (Tải đỉnh — mốc 3)

💥 Kịch bản 3.4: Stress Test tìm điểm gãy (Vượt tải đỉnh)

Tăng dần tải từ 467 VU (7.000 CCU), mỗi bước tương đương +1.000 CCU, duy trì 5 phút/bước để đo trạng thái ổn định:

Bước VU ≈ CCU Duy trì Mục tiêu
14677.0005 phútBaseline (mốc yêu cầu)
25348.0005 phútBắt đầu vượt tải
366710.0005 phútQuan sát suy giảm
483412.5005 phútTìm điểm gãy
51.000+15.000+5 phútXác nhận điểm gãy
Mục tiêu: Xác định chính xác mốc VU nào hệ thống xuất hiện lỗi 502/504 hoặc thời gian phản hồi vượt 10 giây. Điểm đó chính là Điểm gãy của hệ thống.

Tỷ lệ lỗi HTTP≤ 1%Ghi nhận mốc > 10%

Giải thích chỉ số:
• HTTP Duration: Tổng thời gian request (mạng + server)
• Server TTFB: Thời gian server xử lý thuần túy — chỉ số chính xác nhất
• P95: 95% request nhanh hơn giá trị này (loại bỏ outlier)
• P99: 99% request nhanh hơn giá trị này
